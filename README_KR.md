# Valorant Patch Verdict

> 다음 패치에서 누가 너프 / 버프 받을지, 모델이 예측한다.

**라이브 데모**: [whosnxt.app](https://whosnxt.app) · API: [api.whosnxt.app](https://api.whosnxt.app/health)


발로란트를 하다 보면 "이 요원 곧 너프 먹겠다" 하고 감이 오는 순간이 있다. 네온이 VCT에서 77% 픽률을 찍고 있을 때, 제트가 랭크 모든 판에 나올 때, 바이퍼가 프로씬을 지배하는데 랭크에선 아무도 안 쓸 때.

이 프로젝트는 그 "감"을 데이터로 정량화한다. 다이아+ 랭크 통계, VCT 프로 대회 데이터, 역대 패치 이력, 요원 킷 설계를 종합해서 29개 전 요원의 **너프/버프/안정 확률**을 산출한다.

## 모델 구조

### 2단계 분류

```
    요원의 현재 상태 (랭크 통계 + VCT + 패치 이력 + 킷 설계)
                          |
                 [Stage A: XGBoost]  50개 피처
                 "이 요원, 이번에 패치 먹을까?"
                      /          \
                   안정         패치 대상
                                   |
                          [Stage B: XGBoost]  51개 피처
                          "그래서 너프야 버프야?"
                            /          \
                         버프           너프
```

Stage A가 먼저 "패치 압박이 있는가"를 판별하고, 패치 대상으로 잡히면 Stage B가 방향을 결정한다.

두 스테이지는 **서로 다른 피처셋**을 쓴다. "패치를 받느냐"와 "어떤 방향이냐"는 다른 문제이기 때문에, 각각에 최적화된 피처를 따로 골랐다.

| Stage B balanced accuracy | 0.7779 |
| 학습 데이터 | 659행 / 29 요원 / E2A1 ~ V26A2 |

## 핵심 설계

### 픽률 × 승률 2D 사분면

라이엇이 요원을 너프하는 기준은 단순히 "픽률이 높아서" 또는 "승률이 높아서"가 아니다. 레이나가 랭크에서 매판 나와도 승률이 낮으면 안 건드리고(팬덤), 하버가 픽률 0%대인데 이기기만 하면 그것도 안 건드린다(니치). **많이 쓰이면서 동시에 이기는** 요원이 너프 대상이다.

이걸 모델에 반영하기 위해 피처를 2D 평면 위에서 설계했다.

```
                  승률 ↑ (요원 자기 평균 기준)
                        |
     니치 OP            |          너프 대상
     안 쓰는데 이김      |    많이 쓰고 많이 이김
                         |
   ──────────────────────┼──────────── 픽률 → (요원 자기 평균 기준)
                         |
     버프 대상            |          팬덤
     안 쓰고 짐           |    많이 쓰는데 짐
                         |
                  승률 ↓
```

중심점이 핵심이다. 제트의 "정상" 픽률은 14%이고 하버는 0.4%다. 절대 기준으로 자르면 제트는 항상 너프 대상이고 하버는 영원히 버프 대상이 된다. 그래서 **요원 자신의 역대 평균**을 중심으로 잡고, 거기서 얼마나 벗어났는지를 본다.

### 프로씬 고정픽 요원 잡기

바이퍼 문제가 있다. 바이퍼는 역대 VCT 평균 픽률이 이미 40%대다. 상대 비교를 하면 "원래 이 정도인데?" 하고 넘어간다. 하지만 VCT 절반에서 쓰이는 요원이 정상일 리가 없다.

그래서 **VCT 픽률 35% 이상이면 역대 평균과 무관하게 너프 신호**를 건다. 이 절대 기준이 바이퍼/오멘 같은 프로씬 고정 요원을 잡아낸다.

### 신호 유지 (캐리오버)

네온이 이번 액트에 너프 신호가 잡혔는데, 다음 액트까지 패치가 안 나왔다. 수치는 여전히 높다. 그러면 다음 액트에도 너프 신호를 유지한다. 패치 없이 수치가 자연스럽게 내려와서 정상 범위에 들어오면 그때 신호를 해제한다.

이게 없으면 "너프 → 안정 → 너프 → 안정"을 매 액트 반복하는데, 실제로는 패치 압박이 쭉 이어지는 상황이다.

### 레이블 체계

| 레이블 | 의미 |
|---|---|
| `stable` | 수치가 정상 범위. 건드릴 이유 없음 |
| `mild_nerf` | 너프 신호 감지. 아직 패치는 안 나옴 |
| `strong_nerf` | 실제 너프 패치. 핵심 스킬(E/X) 너프 또는 후속 조정 |
| `mild_buff` | 버프 신호 감지. 픽률·승률 다 낮음 |
| `strong_buff` | 실제 버프 패치. 핵심 스킬 버프 또는 리워크 |

strong은 **실제로 패치가 나온 액트**에서만 붙는다. 아무리 수치가 비정상이어도 패치가 안 나왔으면 최대 mild이다.

## 레이블의 뿌리 — 5가지 trigger_type (원래 기획)

처음 기획 단계에서는 라이엇이 왜 특정 요원을 건드리는지를 5가지 trigger로 분류했다. Riot 패치노트의 개발자 코멘트(`change_reason`)를 Claude API로 파싱해 각 변경에 trigger를 붙였고, 이 분류가 현재 5-class 레이블 체계의 뿌리가 됐다.

| trigger_type | 의미 | 실제 분포 (596건) | 대표 사례 |
|---|---|---|---|
| `rank_stat` | 랭크 픽률·승률 기반 조정 | 372건 (62%) | 레이나 Q — 랭크 펍스톰프 억제 |
| `role_invasion` | 역할군 경계 침범 | 104건 (17%) | 킬조이의 공격 능력 과도 → 감시자 본연으로 회귀 |
| `skill_ceiling` | 프로 레벨에서만 과도하게 효과적 | 65건 (11%) | 스카이 Q — 자기 플래시 활용이 "너무 최적화" |
| `pro_dominance` | 프로씬 절대 지배 | 50건 (8%) | 너프 45 vs 버프 5 — 거의 너프 전용 |
| `map_anchor` | 특정 맵에서만 과도하게 강함 | 5건 (1%) | 표본이 작음 |

### 왜 5-class 레이블로 바뀌었나

trigger_type은 **패치가 나온 이유**를 분류한 것이지, **다음 패치를 예측하는 신호**는 아니다. 예를 들어 `pro_dominance`로 너프된 바이퍼를 학습 데이터로 쓸 때, 예측 시점에서는 "바이퍼가 지금 pro_dominance 상태다"를 직접 보기 어렵다. 보이는 건 VCT 픽률 40%뿐이다.

그래서 trigger를 **관측 가능한 지표 기반 피처**로 환원했다:

- `pro_dominance` → `vct_pr_last`, `vct_pr_peak_all` 피처 + VCT 35% 절대 임계
- `role_invasion` → `tier_gap` (킷 설계 등급 vs 실전 성적 괴리), `kit_x_rank_pr`
- `skill_ceiling` → `skill_ceiling_x_vct_pr` (실력 천장 × 프로 픽률)
- `rank_stat` → `rank_pr_excess`, `rank_wr`, `wr_buff_signal`

trigger_type 자체는 **학습 레이블을 만드는 과정**에서 쓰고, 모델은 그 레이블만 맞히도록 학습된다. trigger는 피처가 아닌 **라벨링의 근거**다.

## 사후 판정 → 사전 예측 파이프라인

이 프로젝트의 핵심은 **"이미 나온 패치의 성공/실패를 판정하는 시스템으로 학습 데이터를 만들어, 다음 패치를 예측한다"** 는 2단 구조다.

```
┌──── 1단계: 사후 판정 (verdict) ────────────────────────┐
│  Riot 패치노트   →   Claude API   →   trigger_type     │
│  (change_reason)     (분류)         (5가지)            │
│         ↓                                              │
│  실제 패치 후 지표 변화 측정                            │
│         ↓                                              │
│  combined_verdict:  HIT / MISS / OVERSHOOT /           │
│                     DUAL_MISS / UNDERSHOOT             │
│  (너프했는데 수치 안 변함? MISS.                        │
│   너프했더니 너무 내려감? OVERSHOOT.)                   │
└────────────────────────────────────────────────────────┘
                         ↓
         verdict 이력 → followup/correction 컨텍스트
                         ↓
┌──── 2단계: 사전 예측 (whosnxt.app) ───────────────────┐
│  현재 액트의 지표 + trigger 기반 피처                  │
│         ↓                                             │
│  학습 레이블: stable / mild_nerf / strong_nerf /      │
│              mild_buff / strong_buff                  │
│  (strong은 실제 패치 나온 액트, mild는 신호만 잡힌 액트)│
│         ↓                                             │
│  Stage A (patch 여부) → Stage B (buff/nerf)           │
│         ↓                                             │
│  다음 액트의 p_nerf / p_buff / p_stable 확률 출력      │
└───────────────────────────────────────────────────────┘
```

정리하면:
1. **사후 verdict**로 과거 패치 하나하나에 대해 "이 패치가 효과가 있었나"를 판정
2. **verdict 이력**으로 다음 패치의 컨텍스트 결정 (MISS면 followup, OVERSHOOT면 correction, 둘 다 아니면 first)
3. **현재 지표 + trigger 기반 피처 + 컨텍스트** 로 5-class 레이블 생성 (stable/mild/strong × buff/nerf)
4. 이 레이블을 **XGBoost 타겟**으로 학습시켜 다음 패치를 예측

"사후 판정 엔진"이 **학습 데이터 제조기** 역할을 하고, 그 위에서 사전 예측 모델이 돌아가는 구조다.

## 학습 파이프라인

- **시계열 보호**: Walk-forward CV로 미래 데이터가 과거 평가에 섞이지 않게 한다.
- **Stage A**: stable이 훨씬 많으니까 train fold에서 1:1 언더샘플링. 평가는 원본 비율 그대로.
- **Stage B**: buff가 적으니까 SMOTE로 오버샘플링.
- **하이퍼파라미터**: Optuna TPE Sampler, 스테이지당 60 trials.
- **피처 선택**: 학습 후 SHAP 분석해서 양쪽 다 신호 없는 피처는 빼고, 한쪽에서만 없는 건 그쪽에서만 뺀다.
- **도메인 규칙**: 없다. 모델 출력을 후보정 없이 그대로 쓴다.

## 프론트엔드 페이지 구조

편집자 톤 유지를 위해 홈을 의도적으로 얇게 만들고, 탐색이 필요한 부분은 별도 페이지로 분리했다.

```
/                    TL;DR 2연장 히어로 (Top 너프/버프 1명씩) + 수직 네비 3종
├── /agents          Top 3 너프·버프 그리드 + 전체 로스터 Explorer
├── /agent/[name]    요원 상세 — AI 분석, VCT 타임라인, 패치 이력
├── /simulator       가상 패치 시뮬레이터 (임팩트 + AI 해설)
├── /backtest        Walk-forward 백테스트 리포트
└── /concepts/       시안 검토용 내부 페이지 (예: meta-forecast 스트립 5종)
```

- **홈**(`/`) — `TldrHero` 한 줄 요약 + `NavButton` 세 개(세로 스택). 각 버튼 우측에 요원 초상화를 `objectPosition`으로 크롭해 배경으로 깔고, 요원별 오프셋 오버라이드(`PORTRAIT_Y_OVERRIDE`)로 얼굴 위치를 개별 조정.
- **로스터 분리** — 너프/버프 그리드 + `AgentExplorer`가 홈에서 빠져 `/agents`로 이동. 홈의 편집자 톤과 탐색 기능의 그리드 톤을 섞지 않는다.
- **공통 Back 버튼** — `components/BackToHome.tsx` 하나로 통일(`메인으로` 라벨). 모든 하위 페이지에서 재사용.
- **시안 시트**(`/concepts/meta-forecast`) — 라이브가 아닌 내부 리뷰용. "한 줄 메타 예보" 스트립을 5가지 방향(Split VS, Alert Ticker, Radar Signal, Terminal Readout, Threat Level)으로 동시 비교.

## 현재 한계점

솔직하게 짚고 가는 약점들.

### 신규 요원 데이터 부족

비토(Veto)·믹스(Miks) 같은 2026년 출시 요원은 **패치 이력이 없다.** 모델이 학습한 피처 중 `acts_since_patch`, `last_direction`, `n_nerf_patches` 같은 패치 이력 기반 피처가 대부분 결측이다. 랭크·VCT 지표만으로는 예측 신뢰도가 낮다.

**처리 방식**: `predict_service.py`에서 `acts_since_patch >= 90`이면 방향 무관하게 **stable로 강제**하고 순위 최하단에 배치한다. 상세 페이지에서는 "출시 이후 패치 이력이 없어 예측 신뢰도가 낮습니다"로 표기해 AI 분석도 생략한다. 데이터가 2~3 액트 쌓이면 정상 예측 대상으로 승격한다.

### Claude API의 역할과 한계

Claude는 **학습 파이프라인**과 **런타임 설명** 두 곳에서 쓴다.

| 위치 | 모델 | 역할 | 실패 시 |
|---|---|---|---|
| 학습 데이터 생성 | Claude Sonnet | 패치노트 `change_reason` → `trigger_type` 5-class 분류 | 재시도 · 로그 확인 후 수동 보정 |
| 런타임 요원 설명 | Claude Haiku 4.5 | 예측 근거를 한국어 분석 톤으로 서술 | 5-class 기반 정적 템플릿 폴백 (픽률 구간별 3단계) |
| 시뮬레이터 AI 분석 | Claude Haiku 4.5 | 가상 패치 결과 해설 (역할군별 경쟁 구도 포함) | 에러 메시지 그대로 노출 |

**한계:**
- 학습 단계 trigger 분류는 한 번 돌린 뒤 `patch_notes_classified.csv`로 고정 — 신규 패치마다 갱신 필요.
- 런타임 Haiku는 프롬프트 길이·용어 제한이 많아 할루시네이션 방지에 긴 금지 규칙이 붙어 있음.
- 캐시 오염 가능성: 과거 API 실패 시 템플릿이 캐시돼 남을 수 있어, `cache_key`에 버전 prefix(`v2::`)를 붙여 무효화 가능하게 설계.

### VCT 데이터 시차

현재(2026-04-20) 기준 VCT Stage 1 2026이 진행 중이고, **12.06 패치의 밸런스 변경은 VCT에 4월 24일부터 적용 예정**이다. 즉 프론트에 뜨는 VCT 픽률/승률은 12.06 너프가 반영되지 않은 상태다. 웨이레이처럼 최근 너프된 요원은 현재 VCT 지표가 부풀려져 보일 수 있다.

**처리 방식**: AI 분석 프롬프트에 이 시차를 명시해 "12.06 너프가 VCT에 4월 24일부터 적용될 것" 이라는 컨텍스트를 함께 서술한다. 데이터 테이블에는 `vct_data_lag` 피처로 몇 액트 지연됐는지 정량화해 모델이 가중치를 조절한다.

### 도메인 특성상 근본적인 한계

- **라이엇의 정성적 판단 예측 불가** — "이 요원 킷이 재미없다" 같은 순수 디자인 판단은 데이터에 남지 않음
- **리워크는 예측 난이도 최상** — 사전 신호가 거의 없고, 특정 액트에 갑자기 발표되는 이벤트성 패치
- **2-stage 구조 자체의 에러 전파** — Stage A가 틀리면 Stage B는 실행되지 않아 복구 불가
